# Mini-CRM: Product Requirements Document

**Версия:** 1.0
**Дата:** 2026-04-07
**Статус:** MVP

---

## 1. Problem Statement

Малые и средние розничные магазины ведут учёт клиентов и сделок в Excel-таблицах, мессенджерах или на бумаге. Это приводит к:

- **Потере клиентов**: менеджер уволился — контакты и история общения ушли вместе с ним.
- **Отсутствию контроля**: руководитель не видит, на каком этапе находятся сделки, кто из продавцов работает эффективно, а кто нет.
- **Забытым задачам**: нет системы напоминаний — перезвонить клиенту, отправить КП, подготовить договор.
- **Невозможности анализа**: нет данных по воронке продаж — непонятно, где теряются клиенты.

> Подтверждено типичными болями пользователей CRM-систем в сегменте малого бизнеса: разрозненность данных, отсутствие единого источника правды по клиентам, невозможность контролировать процесс продаж.

## 2. Target Audience

| Характеристика | Описание |
|---------------|----------|
| **Сегмент** | Малый и средний розничный бизнес (1–30 сотрудников в отделе продаж) |
| **Роль: Администратор** | Владелец бизнеса или IT-специалист. Настраивает систему, управляет пользователями и воронкой. Техническая грамотность выше среднего. |
| **Роль: Менеджер** | Руководитель отдела продаж. Контролирует работу продавцов, распределяет клиентов, анализирует отчёты. Нужна полная картина по всем сделкам. |
| **Роль: Продавец (Sales)** | Сотрудник на линии. Ведёт своих клиентов, фиксирует звонки и встречи, двигает сделки по воронке. Нужен простой и быстрый интерфейс. |

---

## 3. Value Proposition

| Критерий | Mini-CRM | Excel / Google Sheets | Bitrix24 / AmoCRM |
|----------|----------|-----------------------|-------------------|
| Стоимость | Бесплатно (self-hosted) | Бесплатно | От 500 руб/мес/пользователя |
| Сложность внедрения | Минимальная (API-first) | Нулевая | Высокая (избыточный функционал) |
| Контроль данных | Полный (свой сервер) | Облако Google/MS | Облако вендора |
| Ролевой доступ | Да (Admin/Manager/Sales) | Нет | Да |
| Воронка продаж | Да, настраиваемая | Нет | Да |
| API для интеграций | REST API с документацией | Нет | Да, но сложный |

**Ключевое отличие**: Mini-CRM — это лёгкая, self-hosted CRM с минимальным порогом входа, без лишних функций, с полным контролем над данными. Для тех, кому enterprise-CRM избыточна, а Excel уже недостаточно.

## 4. MVP Scope

### Must Have (MVP v1.0)

| # | Функция | Описание |
|---|---------|----------|
| 1 | Аутентификация | Регистрация, вход, JWT access/refresh токены |
| 2 | Ролевая модель (RBAC) | Три роли: Admin, Manager, Sales. Разграничение доступа. |
| 3 | Управление пользователями | Admin: CRUD пользователей, назначение ролей, деактивация |
| 4 | CRUD клиентов | Создание, чтение, обновление, удаление. Ownership для Sales. |
| 5 | Сделки с воронкой | CRUD сделок, привязка к клиенту и стадии воронки, автозаполнение closed_at |
| 6 | Настройка стадий воронки | Admin: создание, редактирование, удаление, переупорядочивание стадий |
| 7 | Задачи | CRUD задач по сделкам, назначение ответственного, отметка выполнения |
| 8 | Журнал активностей | Создание записей (звонок, email, встреча, заметка). Лог неизменяем. |
| 9 | Поиск и фильтры | Поиск клиентов по имени/email/компании. Фильтры по сделкам, задачам, активностям. Пагинация. |
| 10 | Мини-отчёт по воронке | Количество и сумма сделок по стадиям, conversion rate |

## 5. User Stories

### Аутентификация

- Как новый пользователь, я хочу зарегистрироваться в системе, чтобы получить доступ к CRM.
- Как пользователь, я хочу войти по email и паролю, чтобы работать с данными.
- Как пользователь, я хочу, чтобы мой сеанс автоматически продлевался через refresh-токен, чтобы не вводить пароль каждые 30 минут.

### Admin

- Как Admin, я хочу создавать и деактивировать пользователей, чтобы управлять доступом к системе.
- Как Admin, я хочу назначать роли (Manager, Sales), чтобы разграничить права доступа.
- Как Admin, я хочу настраивать стадии воронки продаж, чтобы адаптировать CRM под бизнес-процессы магазина.
- Как Admin, я хочу видеть все данные в системе, чтобы иметь полную картину.

### Manager

- Как Manager, я хочу видеть всех клиентов и все сделки, чтобы контролировать процесс продаж.
- Как Manager, я хочу переназначать клиентов и сделки между продавцами, чтобы распределять нагрузку.
- Как Manager, я хочу назначать задачи любому продавцу, чтобы делегировать работу.
- Как Manager, я хочу видеть отчёт по воронке, чтобы понимать, на каких этапах теряются сделки.

### Sales

- Как Sales, я хочу вести своих клиентов (создавать, редактировать), чтобы хранить контактную информацию в одном месте.
- Как Sales, я хочу создавать сделки и двигать их по воронке, чтобы отслеживать прогресс.
- Как Sales, я хочу ставить себе задачи по сделкам с дедлайнами, чтобы не забывать о важных действиях.
- Как Sales, я хочу фиксировать звонки, встречи и заметки в журнале активностей, чтобы сохранять историю общения с клиентом.
- Как Sales, я хочу искать клиентов по имени, email или компании, чтобы быстро находить нужного.
- Как Sales, я хочу видеть отчёт только по своим сделкам, чтобы понимать свою эффективность.

## 6. Non-Functional Requirements

| Требование | Метрика | Обоснование |
|-----------|---------|-------------|
| **Время отклика API** | < 200 мс (p95) | Комфортная работа пользователя без ощутимых задержек |
| **Пропускная способность** | До 100 RPS | Достаточно для 30 одновременных пользователей с запасом |
| **Аутентификация** | JWT access (30 мин) + refresh (7 дней) | Баланс безопасности и удобства |
| **Валидация данных** | Pydantic v2 на входе, CHECK-ограничения в БД | Защита на двух уровнях |
| **Хеширование паролей** | bcrypt (cost factor 12) | Стандарт индустрии |
| **Авторизация** | RBAC с ownership checks | Sales не видит чужие данные |
| **SQL injection** | Параметризованные запросы (SQLAlchemy ORM) | Защита от инъекций |
| **Доступность** | 99% uptime (допустимо для self-hosted MVP) | Нет SLA, локальный сервер |
| **Совместимость БД** | PostgreSQL 14+ | Async-драйвер asyncpg |
| **Документация API** | Автогенерируемая Swagger/OpenAPI | Доступна на `/docs` |
