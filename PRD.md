# PRD – mini-CRM

## 1. Проблема
Малые и средние компании сталкиваются с проблемой управления клиентской базой, сделками и задачами команды продаж. Существующие инструменты либо слишком сложные, либо дорогостоящие, либо не интегрируются с текущим стеком технологий. Цель проекта – предоставить легкую, современную CRM для малого бизнеса с управлением клиентов, сделок, стадий и задач, с базовой аналитикой и безопасным доступом.

## 2. Целевая аудитория
- Малые компании с командой продаж 1–10 человек.
- Менеджеры и sales-агенты, нуждающиеся в простом инструменте для отслеживания сделок.
- Администраторы/руководители, контролирующие работу команды.

## 3. Ценность продукта
- Простая в использовании и развертывании CRM.
- Быстрая настройка и доступ через веб и мобильный браузер.
- Безопасная аутентификация и разграничение ролей.
- Возможность отслеживания прогресса сделок и активности сотрудников.

## 4. MVP (обязательные функции)
- Регистрация и аутентификация пользователей (JWT + refresh token).
- Роли: Admin, Manager, Sales.
- CRUD для клиентов и сделок.
- Управление стадиями сделки.
- Создание и отслеживание задач по клиентам/сделкам.
- Простейшая аналитика: количество клиентов, активных сделок, завершённых задач.
- Docker-развёртывание backend и frontend.
- Unit-тесты для ключевых сервисов.

### Nice to Have (опциональные функции)
- История изменений (audit log).  
- Импорт/экспорт CSV.  
- Напоминания и уведомления.  
- Advanced analytics / отчёты.

## 5. Пользовательские истории
- Как Sales, я хочу создавать новые сделки, чтобы отслеживать прогресс с клиентами.
- Как Manager, я хочу видеть все сделки своей команды, чтобы контролировать эффективность.
- Как Admin, я хочу создавать пользователей и назначать им роли, чтобы управлять доступом.
- Как пользователь, я хочу видеть свои задачи на дашборде, чтобы организовать рабочий день.
- Как Sales, я хочу менять стадию сделки, чтобы фиксировать прогресс.

## 6. Нефункциональные требования
- Время отклика API < 200 мс.
- Нагрузка: поддержка до 100 RPS.
- Безопасность: валидация данных, аутентификация, role-based access control.
- Frontend должен работать на современных браузерах.
- CI/CD для автоматической сборки и тестов.

## 7. Технологический стек
- Backend: Python, FastAPI, PostgreSQL, Alembic, SQLAlchemy.
- Frontend: React, TypeScript, Vite.
- Аутентификация: JWT + refresh token, HTTP-only cookie для refresh.
- Контейнеризация: Docker + Docker Compose.
- Контроль версий: Git (репозиторий на GitHub).

## 8. Этапы реализации
| Этап | Недели | Ключевые вехи | Артефакты |
|------|--------|----------------|------------|
| Discovery | 1–2 | Формирование команды, CustDev, интервью | TEAM.md, Problem Statement, отчёт по интервью |
| Prototyping | 3–4 | Архитектура, дизайн-макеты, CI/CD | Ссылка на макет, схема БД, пустой репозиторий с пайплайном |
| Development | 5–10 | Реализация ядра MVP, 3 спринта | Работающий билд, закрытые тикеты в трекере |
| Stabilization | 11–12 | Тестирование, багфиксы, документация | Отчёт о тестах, инструкция пользователя |
| Pitch Prep | 13 | Подготовка презентации | Питч-дек (10 слайдов) |
| Demo Day | 14 | Финальная защита | Live demo + презентация |

## 9. Критерии приемки
- Все функции MVP работают корректно.
- Тесты проходят без ошибок.
- Docker-контейнеры запускаются без ошибок.
- Frontend и backend корректно интегрированы.
- Code review выполнен, минимум 2 аппрува для мержа в main.
